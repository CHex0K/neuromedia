# Batch Continue On Error Spec

## Original Task

Пользователь заметил риск: при массовой обработке креативов ошибка на одном ролике может прервать весь процесс.

## Acceptance Criteria

AC1. В batch-режиме ошибка обработки одного `.mp4` не должна останавливать обработку следующих `.mp4` файлов из выбранной папки.

AC2. Для каждого упавшего ролика в UI-логе должен быть понятный блок с именем файла, этапом сбоя и traceback/ошибкой.

AC3. Итоговый batch ZIP должен создаваться, если есть хотя бы один успешный ролик, и должен содержать отдельный `batch_failures.json` с информацией обо всех неуспешных роликах.

AC4. Если в batch-режиме упали все ролики, пользователь должен получить понятную ошибку без пустого итогового ZIP.

AC5. Single-file режим должен сохранить прежнее поведение: ошибка одного ролика завершает запуск как `Failed`, чтобы не скрывать проблему.

AC6. Notebook должен оставаться синтаксически валидным JSON, а измененная UI-ячейка должна компилироваться как Python-код.

## Constraints

- Все текстовые файлы и операции чтения/записи должны использовать UTF-8.
- Не менять модельную логику TRIBE, decoder или report generator.
- Не перезаписывать unrelated user changes.
- Не логировать секреты.

## Non-Goals

- Не добавлять параллельную batch-обработку.
- Не добавлять retry всего ролика поверх существующих retry OpenRouter.
- Не менять формат отчетов отдельных успешных роликов.

## Verification Plan

- Проверить текущую UI-ячейку notebook на наличие per-file exception handling в batch mode.
- Проверить, что `batch_failures.json` добавляется в итоговый ZIP.
- Проверить, что notebook JSON читается, а UI-ячейка компилируется.
